============================================================
1) PRODUCT POSITIONING (CURRENT STATE)
============================================================
1.1 Product type
- This product is not "a single chat app."
- It is a dual-entry Agent OS:
  a) IME entry (outside apps)
  b) App entry (inside Lumi main app)

1.2 Core objective
- Solve real user tasks end-to-end first.
- Then continuously improve personalization via digital twin and policy optimization.

1.3 Current delivery style
- Local-first execution with optional cloud enhancement.
- Structured task execution and status tracking, not only free-form chat.

3) END-TO-END USER LOOP (IMPLEMENTED)
============================================================
3.1 Outside app entry path (IME path)
1. User types in any app.
2. User enters Agent Mode.
3. On-device triage computes:
   - requiresCloud
   - score
   - reasonCodes
   - stateVector
4. System decides local vs handoff policy.
5. Complex/high-risk tasks route into Lumi app module flow.

3.2 In-app orchestration path
1. Backend service accepts prompt.
2. Request enters state machine.
3. Orchestrator produces routing + task graph + skills + payload.
4. UI renders module-specific execution cards.
5. User can continue by:
   - follow-up answer
   - context supplement
   - next-action clicks
   - outcome feedback

3.3 Backflow behavior
- Results can flow back to IME candidate area.
- No auto-commit to target input field.
- "Tap to commit" remains enforced.

- When response status is WAITING_USER, Interaction Hub enters pending-question mode.
- Pending question can come from:

  - response.nextAction/summary fallback in WAITING_USER status

- Context supplement path:

- For skill IDs in format trusted:{owner/repo}:{path}:

- Fallback verification URL:
  - https://github.com/anthropics/anthropic-cookbook/tree/main/skills
- This is the implemented "trusted source visible and verifiable" mechanism on Android UI.

============================================================

- Full solve default:
  - nScenarios = 500
  - horizon = 4
- Quick solve default:
  - nScenarios = 100
  - horizon = 2
- Source:
  - core-agent/bellman/BellmanSolver.kt

20) ACCEPTANCE CRITERIA FOR "REAL EXECUTABLE SOLUTION"
============================================================
20.1 Minimum acceptance criteria per run
- A run is accepted as executable only if:
  - output includes concrete next actions
  - constraints are explicit (or pending questions are explicit)
  - evidence references are present for success claims
  - rollback/fallback path is present for risky actions
  - user can continue in-app through Interaction Hub or supplement cards

20.2 Validation checklist for product QA
- Verify WAITING_USER produces visible pending question and submit path.
- Verify user reply changes next orchestration step.
- Verify trusted_catalog skill renders "Anthropic Trusted Skill" and opens verification link.
- Verify Destiny payload includes nextBestAction + alternatives + auditTrail.
- Verify LIX path includes user constraints and does not fabricate missing constraints.
- Verify run history can replay and export module summary.

20.3 Suggested command set for engineering verification
- Unit:
